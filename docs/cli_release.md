# CLI Release Flow

This repo now includes a standalone Dart CLI in `cli/` and a shared headless core in `packages/overlay_core/`.

## Build The macOS CLI Release

From the repo root:

```bash
make build-cli-release-macos
```

That script:

1. builds the frozen Python overlay executables
2. downloads `ffmpeg` and `ffprobe`
3. compiles `cli/bin/fpv_overlay.dart` into a native executable
4. smoke-tests the staged bundle with `--version`, `doctor`, and `render --dry-run`
5. assembles a release bundle with:
   - `bin/fpv-overlay`
   - `runtime/ffmpeg`
   - `runtime/ffprobe`
   - `runtime/osd_overlay/...`
   - `runtime/srt_overlay/...`
6. archives the bundle to `build/cli-release/*.tar.gz`

You can also smoke-test a built bundle manually:

```bash
make smoke-test-cli-bundle ARGS="--archive build/cli-release/fpv-overlay-cli-macos-arm64-<version>.tar.gz --version <version>"
```

## GitHub Actions Release Flow

The repo now includes:

- `.github/workflows/release.yml`

Behavior:

1. On `v*` tag pushes, it builds macOS arm64 and x64 CLI archives.
2. It uploads those archives plus checksum files to the GitHub release for that tag.
3. It renders a filled Homebrew formula with the matching SHA256 values.
4. It uploads that rendered formula as a release asset alongside the desktop artifacts.

You can also run the workflow manually from the Actions tab to produce build artifacts without publishing a GitHub release.

Before pushing a release tag follow the steps in `CLAUDE.md → Release Procedure`, which covers all three version files and the tag flow. Key CLI-specific pre-flight:

1. bump `cli/pubspec.yaml` to the intended release version
2. bump `cli/lib/src/app.dart` `defaultValue` to match
3. run `make verify-cli-release-metadata`
4. push a matching tag such as `v1.0.3`

The CI and release workflows now fail fast if the CLI version and the release tag drift.

## Homebrew Tap Formula

A formula template now lives at:

- `packaging/homebrew/Formula/fpv-overlay.rb`

Before publishing it in your tap repo:

1. upload the CLI release archives to GitHub Releases
2. replace the `url` and `sha256` placeholders in the formula
3. copy the formula into your tap repo, for example:
   - `boonyongyang/homebrew-tap/Formula/fpv-overlay.rb`

If you use the GitHub Actions release workflow, it will generate a ready-to-copy `fpv-overlay.rb` file with the SHA values already filled in.

The formula expects the archive layout generated by `tools/build_cli_release_macos.sh`.

## Suggested Install Command

```bash
brew install boonyongyang/tap/fpv-overlay
```
